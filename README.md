## Project Description

This project implements a complete ETL (Extract, Transform, Load) pipeline for job listings data using Medallion Architecture. The pipeline extracts job data from MHTML files, cleans and processes HTML content, transforms it into structured JSON, loads it into a SQLite database, and performs data quality profiling.

### Module 1: The Extractor (Medallion & Lakehouses)
 Why is it useful to keep the original raw HTML files instead of directly inserting processed data into the database? What problems become easier to debug or recover from?

Keeping raw HTML files provides flexibility, as the data can be reprocessed anytime if the parsing logic changes, without needing to scrape the website again. It also acts as an audit trail, preserving the original source for verification and ensuring data integrity.

Raw HTML makes debugging easier because developers can inspect the original structure to identify parsing errors or missing fields caused by website changes. It also supports recovery, since corrupted or incomplete processed data can be regenerated by reprocessing the saved raw files.

 ### Module 2: Treatment Plant (ETL vs ELT & Scale)
Why do cloud systems prefer loading raw data first before cleaning it (ELT)? What problems happen when processing files sequentially, and how does distributed processing help?

Cloud systems use ELT because loading raw data first allows greater flexibility and scalability. Teams can store large datasets cheaply, then apply different transformations later without needing to reprocess the original data each time.
It also improves efficiency, as transformations can be done using powerful distributed cloud engines, making it faster and more adaptable for different analytics needs.

Sequential processing creates bottlenecks because files are handled one at a time, which becomes slow and inefficient for large datasets. This limits scalability and delays data processing.
Distributed processing solves this by splitting the workload across multiple machines, allowing tasks to run in parallel. This significantly speeds up processing and improves overall system performance.

### Module 3: The Blueprint & The Vault (Storage & Contracts)
What should happen if an important field like job_title disappears? Why fail early instead of silently inserting nulls into DB? How does INSERT OR IGNORE help prevent duplicate records?

If a critical field like job_title is missing, the system should fail early and raise an error instead of inserting incomplete data. This prevents inconsistent or low-quality data from entering the database, which could affect reports and analysis later.
Failing early also makes debugging easier, as developers can quickly identify issues such as parser errors or website structure changes. It helps stop problems at the source before they spread further into the system.

INSERT OR IGNORE helps maintain database consistency by skipping records that already exist based on defined unique constraints. This ensures that duplicate entries are not added during repeated data processing or scraping.
By preventing duplicates, it keeps the database clean and reliable, avoiding redundant data and ensuring each record remains unique and meaningful.

### Module 4: The QA Inspector & Orchestrator (Orchestration & DAGs)
What happens if processor.py crashes halfway? How are automated orchestration tools more reliable than manual retries with Python scripts?

If processor.py crashes midway, it creates confusion about which files were processed. Some JSON files may be generated while others are missing, leaving the dataset incomplete and inconsistent.
Recovery is also manual and inefficient—someone must check logs, figure out where the failure happened, and restart the process correctly, increasing the risk of mistakes.

Airflow improves reliability by tracking each task, so it always knows what has succeeded or failed. It can automatically retry failed tasks, reducing the need for manual intervention.
It also manages task order using DAGs, sends alerts on failures, and can resume from the last successful step, making the workflow more efficient and dependable.
